Phase 3 Reflection Statement (June 24, 2026)

1. How state is saved and restored in the application

In my implementation, the lifecycle for adding loot follows the required sequence of Mutate → Save → updateUI(), which ensures that state remains the single source of truth and that rendering always reflects the latest data. When the user clicks the “Add Loot” button, the addLoot() function validates the inputs and then mutates the state by pushing a new object into the loot array: 

loot.push({ name, value, quantity: qty });

Immediately after this mutation, Phase 3 requires that the updated state be persisted before any rendering occurs. I implemented this by calling saveState() directly after modifying the array: 

saveState();
updateUI();

This ordering is important because it guarantees that the saved state in localStorage always matches the in‑memory state before the UI is rebuilt. The fix was to ensure the correct lifecycle: 

restoreState();
updateUI();

2. Why updateUI() must be called after restoration

During page load, the system must restore state from localStorage before rendering anything. If updateUI() ran before restoreState(), the UI would render an empty loot list even when valid data existed in storage. This actually happened during early testing: I initially placed updateUI() above restoreState(), and the interface always showed “No loot added” until I manually added a new item.

3. What JSON.stringify() and JSON.parse() do conceptually

When I save my application state in Phase 3, I use JSON.stringify() and JSON.parse() to convert data back and forth between JavaScript objects and text stored in localStorage. Conceptually, these two functions act like a pair of translators that allow complex in‑memory structures—such as my loot array and partySize variable—to survive a page reload.

4. What I observed when stepping through the restoration process using breakpoints

During debugging, I set a breakpoint inside saveState() and stepped through the call stack to confirm that the function always executed before updateUI(). This revealed that the lifecycle was functioning correctly: the state object was written to localStorage before the UI attempted to read from the loot array.

5. How this phase supports one or more course outcomes

Phase 3 strengthened my understanding of application lifecycle, state authority, and validation discipline. By implementing persistence responsibly and maintaining architectural boundaries, the system now behaves predictably across reloads while preserving the clean, state‑driven design established in earlier phases.

Phase 2 Logic Explanation (June 21, 2026)

How my loot array is the single source of truth:

  • The loot array holds every item the user adds, including its name, value, and quantity.
  • The interface never stores its own copy of the data. Instead, updateUI() always reads directly from the array.
  • Whenever the array changes—by adding or removing loot—the entire interface updates to match the new state.

How and why I added quantity to the data model:

  • Phase 2 requires total loot to be calculated as value × quantity.
  • Quantity allows the system to represent stacked items realistically (e.g., 3 keys or 5 bars).
  • By storing quantity inside each object, updateUI() can compute accurate totals from the array.

Where updateUI() performs calculations and rendering:

  • updateUI() recalculates the total loot every time the state changes.
  • It rebuilds the loot table by looping through the array and generating rows dynamically.
  • It also updates the Split button, empty‑state message, and visibility of results.
  • No other function is allowed to calculate totals or update the screen, preventing duplicate logic.

How state controls what is visible or enabled:

  • If the loot array is empty, updateUI() shows the “No loot added” message and hides the results.
  • If the party size is invalid, the Split button is disabled even if loot exists.
  • When both loot and party size are valid, the Split button automatically enables.
  • The interface always reflects the current state instead of reacting only to button clicks.

Where conditional logic prevents invalid state:

  • Before adding loot, the system checks that the name is not empty, the value is valid, and the quantity is at least 1.
  • Before splitting loot, the system checks that the party size is valid and that loot exists.
  • These checks prevent bad data from entering the array and ensure updateUI() always works with clean state.
